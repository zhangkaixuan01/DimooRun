# Product Workflow Coverage Matrix

This matrix reconciles the production gap plan against the current implementation. It intentionally treats generic collection CRUD as `partial` unless the workflow has domain validation, action availability, audit behavior, success feedback, failure recovery, and browser evidence.

Status values:

- `complete`: current code and tests prove the workflow expectation at production-workflow level.
- `partial`: some routes, models, pages, or tests exist, but workflow semantics are incomplete.
- `missing`: no meaningful implementation exists for the workflow expectation.

## Design Guardrail Mapping

| Guardrail | Workflows it constrains | Rule |
|---|---|---|
| Runtime Control Plane | All workflows | Workflows must manage, observe, govern, and recover runtime behavior instead of becoming a builder surface. |
| Business black box / runtime white box | Runs, replay, observability, quality, audit | Do not inspect or mutate business semantics beyond user-provided evidence; expose runtime state, attempts, events, artifacts, traces, policy, and cost. |
| No low-code builder | Agent package, deployment, catalog/assets | Do not introduce drag-and-drop agent construction or business workflow canvas. Users bring agent code. |
| No prompt designer | Prompt assets, quality, replay | Prompt assets are governed runtime assets, not a prompt creation product. |
| No workflow builder | Scheduled/batch, deployment, gateway | Scheduling and routing are runtime control shapes, not a general workflow automation product. |
| No Policy Engine bypass | Governance, tools, model gateway, secrets, identity, settings | Dangerous actions must go through permission summary, policy decision, audit behavior, and backend enforcement. |

## Completion Rule

When a future phase changes a verdict, update this matrix in the same change. A workflow cannot move from `partial` to `complete` unless the corresponding acceptance mapping has implementation evidence and browser evidence.
